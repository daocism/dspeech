# Dspeech north-star best practices - 2026-05-28

Status: authoritative research reference for downstream Dspeech leads. Read this before code or copy changes.

Source note: the worker runtime did not expose Context7 tools. The research DAG used primary upstream sources only: Apple Developer Documentation, Swift.org, Swift Evolution proposals, Apple WWDC pages, and upstream open-source repositories. Repo-local rules cite file lines from `feat/local-pilot-voice-filter`. Unless marked as a source gap, every recommendation below has two or more supporting sources, or is a repo-local rule grounded in `CLAUDE.md` / ADRs / code.

## 0. Binding repo rules

1. Local-only remains the default. No code path may send audio, transcripts, or derived metadata off-device while cloud privacy is disabled. The privacy badge must remain visible. Sources: `CLAUDE.md:14-19`, `docs/adr/0002-privacy-local-only-default.md:13-27`, `Dspeech/Core/Settings/PrivacySettings.swift:4-24`, `Dspeech/App/ContentView.swift:216-231`.
2. The implementation is protocol-first: SwiftUI composes dependencies, service protocols own I/O boundaries, and state owners are `@MainActor @Observable`. Sources: `CLAUDE.md:25-31`, `Dspeech/App/ContentView.swift:3-41`, `Dspeech/App/CaptureCoordinator.swift:1-88`, `Dspeech/App/LiveTranscriptionViewModel.swift:1-97`.
3. Voice filtering must fail open. Only confident pilot/self audio may be discarded; disabled state, missing profiles, mixed speech, insufficient data, missing model packs, unavailable identifiers, and classifier errors continue to ASR. Sources: `Dspeech/Core/VoiceFilter/VoiceFilterPipeline.swift:157-172`, `Dspeech/Core/VoiceFilter/VoiceFilterPipeline.swift:202-215`, `Dspeech/Core/ASR/SerialBufferRouter.swift:43-63`, `Dspeech/Core/ASR/UtteranceWindowRouter.swift:47-56`.
4. App Store metadata, TestFlight submission, billing, paid acquisition, hardware purchases, and cloud-network paths require explicit Andrei sign-off. Sources: `CLAUDE.md:21-23`, `CLAUDE.md:83-90`, `docs/adr/0006-go-to-market-no-call-script.md:13-27`.
5. Do not promise flight-safety, certified cockpit reliability, or validated hardware support. Sources: `docs/adr/0008-local-speaker-model-pack-readiness.md:72-76`, `docs/eval/local-speaker-model-pack-validation.md:7-9`, `docs/adr/0004-no-hardware-purchase-cable-testing.md:11-22`.

Important drift to carry forward: ADR 0007 is stale for the active branch. It says not to import FluidAudio until later gates, but the current branch already pins FluidAudio, implements `FluidAudioSpeakerIdentifier`, and keeps execution behind the installed model-pack gate. Treat ADR 0008 plus current code as the active contract. Sources: `docs/adr/0007-voice-filter-phase-1-callsign-only.md:36-45`, `Dspeech/Core/VoiceFilter/FluidAudioSpeakerIdentifier.swift:1-174`, `Dspeech/Core/VoiceFilter/SpeakerModelPackInstaller.swift:1-42`, `Dspeech.xcodeproj/project.xcworkspace/xcshareddata/swiftpm/Package.resolved:1-15`.

## 1. Swift 6 strict concurrency in production iOS apps

Use Swift 6 strict concurrency as an architecture constraint, not as a lint cleanup.

Recommendations:

- Keep all user-visible mutable state in `@MainActor @Observable` owners. Dspeech already follows this in `CaptureCoordinator`, `LiveTranscriptionViewModel`, and `PrivacySettings`. This aligns with Swift global actors and Apple Observation guidance. Sources: `Dspeech/App/CaptureCoordinator.swift:1-88`, `Dspeech/App/LiveTranscriptionViewModel.swift:1-97`, [SE-0316 Global Actors](https://github.com/swiftlang/swift-evolution/blob/main/proposals/0316-global-actors.md), [Observation `Observable`](https://developer.apple.com/documentation/observation/observable).
- Keep classifier, audio-buffer conversion, and model work off the main actor. Dspeech's `SerialBufferRouter`, `UtteranceWindowRouter`, and `FluidAudioDiarizerHandle` are the canonical seams. Sources: `Dspeech/Core/ASR/SerialBufferRouter.swift:1-66`, `Dspeech/Core/ASR/UtteranceWindowRouter.swift:1-69`, `Dspeech/Core/VoiceFilter/FluidAudioSpeakerIdentifier.swift:46-73`, [SE-0338 non-actor-isolated async execution](https://github.com/swiftlang/swift-evolution/blob/main/proposals/0338-clarify-execution-non-actor-async.md).
- Domain values that cross actor boundaries must be `Sendable`; storage adapters that wrap SDK types need explicit review before `@unchecked Sendable`. Sources: `Dspeech/Core/Models/TranscriptSegment.swift:3-8`, `Dspeech/Core/VoiceFilter/ModelPackState.swift:1-152`, [SE-0302 `Sendable` and `@Sendable`](https://github.com/swiftlang/swift-evolution/blob/main/proposals/0302-concurrent-value-and-concurrent-closures.md).
- Use `nonisolated` deliberately. For pure conversion helpers, it is acceptable; for stateful service methods, prefer actor or protocol isolation. Sources: `Dspeech/Core/ASR/AppleSpeechLiveTranscriptionEngine.swift:187-228`, [SE-0313 Improved control over actor isolation](https://github.com/swiftlang/swift-evolution/blob/main/proposals/0313-actor-isolation-control.md), [SE-0338](https://github.com/swiftlang/swift-evolution/blob/main/proposals/0338-clarify-execution-non-actor-async.md), [SE-0461 async function isolation](https://github.com/swiftlang/swift-evolution/blob/main/proposals/0461-async-function-isolation.md).
- Treat region-based isolation as a reason to model ownership precisely, not as permission to pass mutable shared state around. Sources: [SE-0414 Region Based Isolation](https://github.com/swiftlang/swift-evolution/blob/main/proposals/0414-region-based-isolation.md), [Swift 6 concurrency migration guide](https://www.swift.org/migration/documentation/swift-6-concurrency-migration-guide/incrementaladoption/).
- Actor reentrancy rule: no actor method may assume state remains unchanged across `await`. Snapshot inputs before suspension, then re-check gates after suspension. This is required for capture state, model-pack state, and privacy state. Sources: [SE-0306 Actors](https://github.com/swiftlang/swift-evolution/blob/main/proposals/0306-actors.md), `Dspeech/Core/ASR/SerialBufferRouter.swift:1-66`.
- Swift proposal pins for this repo: SE-0302 (`Sendable`), SE-0306 (actors and reentrancy), SE-0313 (`nonisolated` actor isolation control), SE-0316 (global actors / `MainActor`), SE-0337 (strict concurrency migration), SE-0338 (non-actor async execution), SE-0414 (region-based isolation), SE-0449 (`nonisolated` global-actor cutoff), SE-0461 (async function isolation). Sources: [SE-0337](https://github.com/swiftlang/swift-evolution/blob/main/proposals/0337-support-incremental-migration-to-concurrency-checking.md), [SE-0449](https://github.com/swiftlang/swift-evolution/blob/main/proposals/0449-nonisolated-for-global-actor-cutoff.md).

## 2. SwiftUI architecture

Recommendations:

- Use `@Observable` for app and view models; do not introduce new `ObservableObject` / `@Published` unless a dependency forces interoperability. Sources: [Migrating from `ObservableObject` to `@Observable`](https://developer.apple.com/documentation/swiftui/migrating-from-the-observable-object-protocol-to-the-observable-macro), [`Observable`](https://developer.apple.com/documentation/observation/observable), [Discover Observation in SwiftUI](https://developer.apple.com/videos/play/wwdc2023/10149/), `CLAUDE.md:25-31`.
- Use `@State` only for values the view owns. `ContentView` owns the coordinator, voice-filter pipeline, and privacy model; it injects services rather than performing I/O in view bodies. Sources: `Dspeech/App/ContentView.swift:3-41`, [`State`](https://developer.apple.com/documentation/swiftui/state).
- Prefer explicit initializer injection for services and fakes. Use environment only for cross-cutting UI dependencies that are already environment-shaped. This keeps test construction deterministic. Sources: `Dspeech/App/ContentView.swift:10-41`, `DspeechTests/LiveTranscriptionViewModelTests.swift:1-197`, [`Environment`](https://developer.apple.com/documentation/swiftui/environment).
- Keep view-model isolation on `@MainActor`. Long-running capture, classifier, and file/model-pack work must sit behind protocols, actors, or async services. Sources: `Dspeech/App/LiveTranscriptionViewModel.swift:1-97`, `Dspeech/Core/Audio/AudioSessionRouting.swift:1-78`, `Dspeech/Core/VoiceFilter/LocalSpeakerIdentifier.swift:65-102`.
- Accessibility identifiers are test infrastructure. Add them with every UI control that XCUITest needs. Sources: `CLAUDE.md:29-31`, `DspeechUITests/DspeechUITests.swift:1-138`.

## 3. On-device speech recognition

Target shape:

- Primary ASR path for iOS 26+ is Apple's `SpeechAnalyzer` with `SpeechTranscriber`. Use `DictationTranscriber` only as fallback for devices/locales where the richer transcriber is unavailable. Sources: [Speech framework docs](https://developer.apple.com/documentation/speech/), [`SpeechAnalyzer`](https://developer.apple.com/documentation/speech/speechanalyzer), [`SpeechTranscriber`](https://developer.apple.com/documentation/speech/speechtranscriber), [`DictationTranscriber`](https://developer.apple.com/documentation/speech/dictationtranscriber), [WWDC25 SpeechAnalyzer session](https://developer.apple.com/videos/play/wwdc2025/277/).
- Enforce local-only semantics at the service boundary. For the current `SFSpeechRecognizer` path, `requiresOnDeviceRecognition = true` is mandatory. No fallback may silently re-enable cloud ASR. Sources: `Dspeech/Core/ASR/AppleSpeechLiveTranscriptionEngine.swift:93-100`, `docs/adr/0002-privacy-local-only-default.md:13-27`, [`SFSpeechRecognitionRequest.requiresOnDeviceRecognition`](https://developer.apple.com/documentation/speech/sfspeechrecognitionrequest/3152603-requiresondevicerecognition).
- Locale/model lifecycle must be explicit. For Apple speech modules, check `AssetInventory.status(forModules:)`; if installation is needed, `AssetInventory.assetInstallationRequest(supporting:)` and `downloadAndInstall()` are allowed only behind explicit Andrei-approved, user-initiated install UX with source/size disclosure. Never install Apple speech assets on app launch or capture start, and never start capture while required assets are unsupported or downloading. These Apple-managed assets come from Apple servers and are retained, updated, and shared by the system, so they inherit the same network-signoff, privacy-badge, and post-install offline verification gates as Dspeech model packs. For FluidAudio, use Dspeech's separate user-gated installed-pack state, local `.mlmodelc` checks, and post-install offline gate. Sources: [`SpeechAnalyzer`](https://developer.apple.com/documentation/speech/speechanalyzer), [`AssetInventory`](https://developer.apple.com/documentation/speech/assetinventory), [`AssetInventory.status(forModules:)`](https://developer.apple.com/documentation/speech/assetinventory/status%28formodules%3A%29), [`assetInstallationRequest(supporting:)`](https://developer.apple.com/documentation/speech/assetinventory/assetinstallationrequest%28supporting%3A%29), [`downloadAndInstall()`](https://developer.apple.com/documentation/speech/assetinstallationrequest/downloadandinstall%28%29), `CLAUDE.md:16-21`, `CLAUDE.md:83-90`, `docs/adr/0008-local-speaker-model-pack-readiness.md:35-55`.
- Compare responsibilities correctly. Apple SpeechAnalyzer handles ASR; FluidAudio handles VAD, diarization primitives, and enrolled speaker filtering; whisper.cpp is local ASR and does not solve enrolled pilot filtering. Sources: [FluidAudio](https://github.com/FluidInference/FluidAudio), [FluidAudio diarization guide](https://docs.fluidinference.com/diarization/getting-started.md), [FluidAudio speaker manager](https://docs.fluidinference.com/diarization/speaker-manager.md), [FluidAudio VAD guide](https://docs.fluidinference.com/vad/getting-started.md), [whisper.cpp](https://github.com/ggml-org/whisper.cpp), `Dspeech/Core/VoiceFilter/FluidAudioSpeakerIdentifier.swift:1-174`.
- FluidAudio is the MVP speaker-filter backend, but only behind installed-pack state and local `.mlmodelc` checks. Its code is Apache-2.0; model attribution still needs final packaging review. Sources: [FluidAudio repo](https://github.com/FluidInference/FluidAudio), [FluidAudio Package.swift](https://raw.githubusercontent.com/FluidInference/FluidAudio/main/Package.swift), [FluidAudio LICENSE](https://raw.githubusercontent.com/FluidInference/FluidAudio/main/LICENSE), [FluidAudio v0.14.7 release](https://github.com/FluidInference/FluidAudio/releases/tag/v0.14.7), [speaker diarization model card](https://huggingface.co/FluidInference/speaker-diarization-coreml), `Dspeech/Core/VoiceFilter/SpeakerModelPackInstaller.swift:1-42`.
- Do not use Picovoice Eagle for this local-only MVP despite feature relevance: the iOS binding requires an AccessKey and binary target; that fails this repo's source and privacy posture. Sources: [Picovoice Eagle repo](https://github.com/Picovoice/eagle), [Picovoice Eagle iOS README](https://raw.githubusercontent.com/Picovoice/eagle/master/binding/ios/README.md), [Picovoice Eagle Package.swift](https://raw.githubusercontent.com/Picovoice/eagle/master/Package.swift), `CLAUDE.md:14-19`.

## 4. Privacy manifests and App Store privacy answers

Separate these artifacts:

- `PrivacyInfo.xcprivacy` describes app/SDK data practices and required-reason API usage in the bundle. Sources: [Privacy manifest files](https://developer.apple.com/documentation/bundleresources/privacy_manifest_files), [Describing data use in privacy manifests](https://developer.apple.com/documentation/bundleresources/describing-data-use-in-privacy-manifests), [Describing use of required reason API](https://developer.apple.com/documentation/bundleresources/describing-use-of-required-reason-api), [WWDC23 privacy manifests](https://developer.apple.com/videos/play/wwdc2023/10060/).
- App Store privacy details are the public privacy-label answers in App Store Connect. Local on-device processing is not data collection by itself; any transmitted analytics, logs, account, support, purchase, or crash data must be declared. Sources: [App privacy details](https://developer.apple.com/app-store/app-privacy-details/), [App Store Connect help](https://developer.apple.com/help/app-store-connect/).
- Every third-party SDK or binary dependency must be audited for its own manifest, tracking domains, and required-reason APIs before it enters the app target. Sources: [Third-party SDK requirements](https://developer.apple.com/support/third-party-SDK-requirements/), [Privacy manifest files](https://developer.apple.com/documentation/bundleresources/privacy_manifest_files), [FluidAudio Package.swift](https://raw.githubusercontent.com/FluidInference/FluidAudio/main/Package.swift).
- Do not add analytics or logging that captures audio, transcripts, speaker embeddings, callsigns, route metadata, or model-pack paths under local-only mode. Sources: `CLAUDE.md:14-19`, `docs/adr/0002-privacy-local-only-default.md:13-27`, `docs/adr/0008-local-speaker-model-pack-readiness.md:35-42`.
- Treat privacy labels as App Store Connect metadata requiring sign-off. The research DAG did not find a primary Apple REST endpoint for editing the privacy-label questionnaire after checking App Store Connect API search results for privacy data/app privacy details/app data usages; do not automate that write path until a later docs refresh proves the endpoint and payload. Sources: [App privacy details](https://developer.apple.com/app-store/app-privacy-details/), [App Store Connect help](https://developer.apple.com/help/app-store-connect/), `CLAUDE.md:83-90`.

## 5. Swift Testing vs XCTest

Policy:

- Use Swift Testing for domain, router, model-pack state, classifier, and service-boundary tests. Sources: [`Testing`](https://developer.apple.com/documentation/testing), [Defining tests](https://developer.apple.com/documentation/testing/definingtests), [Traits](https://developer.apple.com/documentation/testing/traits), [WWDC24 Swift Testing](https://developer.apple.com/videos/play/wwdc2024/10179/), `CLAUDE.md:29-31`.
- Keep XCTest/XCUITest for UI smoke and app-launch flows. Sources: `CLAUDE.md:29-31`, `DspeechUITests/DspeechUITests.swift:1-138`.
- Prefer `@Test`, `#expect`, `@Suite`, async tests, actor-isolated tests, parameterized inputs, tags/traits, and deterministic fakes. Sources: [`Testing`](https://developer.apple.com/documentation/testing), [Defining tests](https://developer.apple.com/documentation/testing/definingtests), `DspeechTests/SerialBufferRouterTests.swift:96-244`, `DspeechTests/LiveTranscriptionViewModelTests.swift:1-197`.
- Do not force UI tests into Swift Testing. Use XCUITest where the assertion is about the running app, accessibility identifiers, permission surfaces, or Settings flows. Sources: [XCTest](https://developer.apple.com/documentation/xctest), [Swift Testing](https://developer.apple.com/documentation/testing), `DspeechUITests/DspeechUITests.swift:1-138`, `CLAUDE.md:29-31`.
- Parity gap to respect: Swift Testing is not a replacement for XCUITest automation. It is the default for logic; XCTest remains the app/UI harness. Sources: [XCTest](https://developer.apple.com/documentation/xctest), [Swift Testing](https://developer.apple.com/documentation/testing), `DspeechUITests/DspeechUITests.swift:1-138`.

## 6. Property-based testing in Swift

Recommendation: use `x-sheep/swift-property-based` (`PropertyBased`) for Dspeech voice-filter classifier properties.

Why:

- It is Swift Testing-native, Swift 6-oriented, has generators, shrinking, fixed-seed reproduction, and no runtime dependency surface for the app target. Sources: [PropertyBased repo](https://github.com/x-sheep/swift-property-based), [PropertyBased Package.swift](https://raw.githubusercontent.com/x-sheep/swift-property-based/main/Package.swift), [PropertyBased release 1.2.0](https://github.com/x-sheep/swift-property-based/releases/tag/1.2.0), [PropertyBased license](https://raw.githubusercontent.com/x-sheep/swift-property-based/main/LICENSE).
- It fits the classifier risks: threshold monotonicity, fail-open decisions, embedding vector dimension invariants, route ordering, and "only confident pilot means discard" should be properties, not just examples. Sources: `Dspeech/Core/VoiceFilter/VoiceFilterPipeline.swift:157-215`, `Dspeech/Core/ASR/SerialBufferRouter.swift:1-66`, `Dspeech/Core/ASR/UtteranceWindowRouter.swift:1-69`.
- Keep it test-only. Do not add it to app targets. Sources: `CLAUDE.md:25-31`, [PropertyBased Package.swift](https://raw.githubusercontent.com/x-sheep/swift-property-based/main/Package.swift).

Rejected for this repo: SwiftCheck is older and XCTest-era, while PropertyBased fits the pinned Swift Testing direction. Sources: [SwiftCheck](https://github.com/typelift/SwiftCheck), [PropertyBased repo](https://github.com/x-sheep/swift-property-based).

## 7. Audio capture and route-change handling

Recommendations:

- Request microphone permission through `AVAudioApplication`; keep `AVAudioSession.requestRecordPermission` only as a compatibility fallback where required. Sources: [`AVAudioApplication.requestRecordPermission`](https://developer.apple.com/documentation/avfaudio/avaudioapplication/requestrecordpermission%28completionhandler%3A%29), [`AVAudioSession.requestRecordPermission`](https://developer.apple.com/documentation/avfaudio/avaudiosession/requestrecordpermission%28_%3A%29), `Dspeech/Core/ASR/AppleSpeechLiveTranscriptionEngine.swift:280-289`.
- Configure `AVAudioSession` before capture, validate the input format, install a tap on the input node, and remove taps during cleanup. Sources: [`AVAudioSession`](https://developer.apple.com/documentation/avfaudio/avaudiosession), [`AVAudioEngine.inputNode`](https://developer.apple.com/documentation/avfaudio/avaudioengine/inputnode), [`AVAudioNode.installTap`](https://developer.apple.com/documentation/avfaudio/avaudionode/installtap%28onbus%3Abuffersize%3Aformat%3Ablock%3A%29), `Dspeech/Core/ASR/AppleSpeechLiveTranscriptionEngine.swift:45-100`, `Dspeech/Core/ASR/AppleSpeechLiveTranscriptionEngine.swift:129-136`.
- Route changes and interruptions are product events, not logs. Wrap `AVAudioSession` behind `AudioSessionRouting`, classify route health in pure code, and update UI through `@MainActor`. Sources: [Responding to audio route changes](https://developer.apple.com/documentation/avfaudio/responding_to_audio_route_changes), [Handling audio interruptions](https://developer.apple.com/documentation/avfaudio/handling_audio_interruptions), `Dspeech/Core/Audio/AudioSessionRouting.swift:1-78`, `Dspeech/Core/Audio/LiveAudioSessionRouting.swift:1-101`, `Dspeech/Core/Audio/RouteHealthClassifier.swift:1-59`, `Dspeech/App/CaptureCoordinator.swift:36-88`.
- AVAudioEngine tap closures must not mutate UI state directly. Convert/submit buffers through a safe boundary, then serialize append/discard decisions through the router. Sources: `Dspeech/Core/ASR/AppleSpeechLiveTranscriptionEngine.swift:103-136`, `Dspeech/Core/ASR/SerialBufferRouter.swift:1-66`, `Dspeech/Core/ASR/UtteranceWindowRouter.swift:1-69`.
- The current route classifier treats some wireless/external routes as suitable. Do not turn that into a support promise; ADR 0004 still keeps wireless validation outside the current support story. Sources: `docs/adr/0004-no-hardware-purchase-cable-testing.md:15-17`, `Dspeech/Core/Audio/RouteHealthClassifier.swift:32-38`, `DspeechTests/RouteHealthClassifierTests.swift:176-192`.

## 8. TestFlight and App Store Connect API

Policy:

- Do not submit, update metadata, invite external testers, or publish privacy answers from CI without explicit sign-off. Sources: `CLAUDE.md:21-23`, `CLAUDE.md:83-90`, `docs/adr/0006-go-to-market-no-call-script.md:13-27`.
- Signing/upload path: prefer Xcode or Transporter for this repo unless a later signed-off automation slice explicitly adopts Apple's Build Uploads API. App Store Connect API also exposes build-upload resources, so do not claim the API is metadata-only. Sources: [App Store Connect API builds](https://developer.apple.com/documentation/appstoreconnectapi/builds), [Build Uploads](https://developer.apple.com/documentation/appstoreconnectapi/build-uploads), [App Store Connect help](https://developer.apple.com/help/app-store-connect/).
- Automation credential gate: no App Store Connect API key, private key, JWT material, Transporter credential, signing certificate, provisioning profile, or exported signing artifact may be committed, printed, or stored in docs. Use least-privilege ASC keys, approved secret storage (`op://` in this fleet), CI log redaction, signing-asset separation from upload automation, and written rotation/revocation steps before any Transporter or ASC automation is enabled. Sources: [Creating API keys for App Store Connect API](https://developer.apple.com/documentation/appstoreconnectapi/creating-api-keys-for-app-store-connect-api), [Create a certificate signing request](https://developer.apple.com/help/account/certificates/create-a-certificate-signing-request), [Create a development provisioning profile](https://developer.apple.com/help/account/provisioning-profiles/create-a-development-provisioning-profile), `CLAUDE.md:83-90`.
- TestFlight path: internal group first; external testing requires Apple beta review metadata and review submission. Sources: [TestFlight overview](https://developer.apple.com/help/app-store-connect/test-a-beta-version/testflight-overview/), [Add internal testers](https://developer.apple.com/help/app-store-connect/test-a-beta-version/add-internal-testers/), [Invite external testers](https://developer.apple.com/help/app-store-connect/test-a-beta-version/invite-external-testers/), [Provide test information](https://developer.apple.com/help/app-store-connect/test-a-beta-version/provide-test-information), [`betaAppReviewSubmissions`](https://developer.apple.com/documentation/appstoreconnectapi/beta-app-review-submissions).
- App Store Connect API resources to know: `GET /v1/betaGroups`, build-to-beta-group relationships, `betaAppReviewSubmissions`, `appEncryptionDeclarations`, and `apps`. Use current Apple docs for exact payloads. Sources: [`beta-groups`](https://developer.apple.com/documentation/appstoreconnectapi/beta-groups), [`betaAppReviewSubmissions`](https://developer.apple.com/documentation/appstoreconnectapi/beta-app-review-submissions), [`appEncryptionDeclarations`](https://developer.apple.com/documentation/appstoreconnectapi/app-encryption-declarations), [`apps`](https://developer.apple.com/documentation/appstoreconnectapi/apps).
- Export compliance: set and verify `ITSAppUsesNonExemptEncryption` only after reviewing the actual cryptography surface. Speech/audio processing itself is not the encryption declaration; networking, transport security, and bundled crypto are. Sources: [Overview of export compliance](https://developer.apple.com/help/app-store-connect/manage-app-information/overview-of-export-compliance), [`appEncryptionDeclarations`](https://developer.apple.com/documentation/appstoreconnectapi/app-encryption-declarations).
- Privacy nutrition labels: prepare answers from the privacy matrix, but treat final submission as sign-off work. Sources: [App privacy details](https://developer.apple.com/app-store/app-privacy-details/), `docs/adr/0002-privacy-local-only-default.md:13-27`, `docs/adr/0008-local-speaker-model-pack-readiness.md:35-55`.

Release gate for Dspeech TestFlight: local build/test lanes, model-pack offline validation, replay/source-audio evidence, privacy copy, signing, export compliance, and App Store privacy answers must all be green. Sources: `docs/adr/0005-app-first-cockpit-validation-later.md:15-29`, `docs/adr/0008-local-speaker-model-pack-readiness.md:43-55`, `docs/eval/local-speaker-model-pack-validation.md:66-86`, `docs/ai-kb/current-context.md:38-45`.

## 9. Adversarial review checklist for AI-authored iOS code

Block until each item is answered with source lines, tests, or a stated gap:

- Audio buffer off-by-one: sample counts, frame lengths, channel conversion, and stereo averaging need explicit review. Current tests cover mono extraction, stereo averaging, non-float rejection, and empty buffers; boundary and oversized-buffer coverage should be treated as a gap until added. Sources: `Dspeech/Core/ASR/AppleSpeechLiveTranscriptionEngine.swift:187-228`, `DspeechTests/VoiceFilterTests.swift:1085-1119`, `DspeechTests/UtteranceWindowRouterTests.swift:6-67`.
- `@MainActor` violations: no heavy classification, model loading, or file I/O on the main actor. Sources: `Dspeech/Core/ASR/SerialBufferRouter.swift:1-66`, `Dspeech/Core/VoiceFilter/FluidAudioSpeakerIdentifier.swift:46-73`.
- Stale closures: tap, recognition-task, route-change, and observation callbacks must use weak captures or explicit lifecycle ownership, and must not emit after stop/finish. Sources: `Dspeech/Core/ASR/AppleSpeechLiveTranscriptionEngine.swift:117-163`, `Dspeech/Core/ASR/SerialBufferRouter.swift:56-66`, `Dspeech/App/CaptureCoordinator.swift:36-88`.
- Retain cycles: any long-lived `Task`, notification sequence, or engine callback must show cancellation/cleanup ownership. Sources: `Dspeech/App/LiveTranscriptionViewModel.swift:81-97`, `Dspeech/Core/ASR/AppleSpeechLiveTranscriptionEngine.swift:230-250`.
- ARC vs Swift isolation: `@unchecked Sendable` must be justified by immutable state, injected synchronization, or SDK thread-safety. Sources: `Dspeech/Core/Settings/PrivacySettings.swift:27-75`, `Dspeech/Core/Audio/AudioSessionRouting.swift:1-78`.
- Fail-open invariant: classifier errors and ambiguous speaker decisions must append audio to ASR, not drop it. Sources: `Dspeech/Core/VoiceFilter/VoiceFilterPipeline.swift:157-215`, `Dspeech/Core/ASR/SerialBufferRouter.swift:43-63`.
- Privacy invariant: grep for `URLSession`, `Network.framework`, telemetry, uploads, analytics SDKs, and model auto-download paths in any local-only feature branch. Sources: `CLAUDE.md:14-19`, `docs/adr/0008-local-speaker-model-pack-readiness.md:35-42`.
- Model-pack gate: every FluidAudio call path must prove installed pack, local model path, required model files, embedding dimension, and fail-open fallback. Sources: `Dspeech/Core/VoiceFilter/FluidAudioSpeakerIdentifier.swift:139-174`, `Dspeech/Core/VoiceFilter/LocalSpeakerIdentifier.swift:65-102`, `DspeechTests/VoiceFilterTests.swift:1159-1235`.
- UI truthfulness: no fake buttons, disabled retry with empty action, "coming soon" copy, or fake cloud/AI/transcription surfaces. Sources: `CLAUDE.md:17-18`, `docs/adr/0008-local-speaker-model-pack-readiness.md:57-62`, `Dspeech/App/ContentView.swift:700-718`.

## 10. Anti-patterns banned for this repo

- Fake cloud, fake AI, fake transcription, placeholder UI, or demo behavior that looks real. Sources: `CLAUDE.md:17-18`, `Dspeech/Core/Models/TranscriptSegment.swift:3-8`.
- Silent cloud fallback, telemetry, model download, analytics, or support logging under local-only mode. Sources: `CLAUDE.md:14-19`, `docs/adr/0002-privacy-local-only-default.md:13-27`, `docs/adr/0008-local-speaker-model-pack-readiness.md:35-42`.
