# Accessibility audit — 2026-05-19

Scope: every SwiftUI source file in `Dspeech/App/` (5 files) plus
user-visible strings in `Dspeech/Core/Settings/PrivacySettings.swift`.
Read-only audit; no source changed. Findings cite `Dspeech/<path>:<line>`.

Severity tags: **BLOCKER** (VoiceOver-only or AX5 user cannot operate the
feature), **SHOULD-FIX** (degraded but operable), **NICE** (polish).

UITest contract this audit is consistent with: identifiers used in
`DspeechUITests/DspeechUITests.swift` (`translation-toggle`,
`settings-button`, `settings-done-button`, `cloud-toggle`,
`privacy-badge`, `Dspeech` title) are preserved — none of the
recommendations below propose renaming or removing those anchors.

## 1. Missing or weak labels / identifiers on user-visible elements

- **SHOULD-FIX** `Dspeech/App/ContentView.swift:95` — `Image(systemName:
  liveViewModel.isListening ? "stop.fill" : "mic.fill")` sits inside the
  Start/Stop button next to a `Text` label and is not marked
  `.accessibilityHidden(true)`. VoiceOver reads "microphone, Старт",
  double-announcing intent.
- **SHOULD-FIX** `Dspeech/App/ContentView.swift:108` — Start/Stop button
  has an identifier but no `accessibilityLabel`/`accessibilityValue`
  describing recording state. A blind user cannot tell from the trait
  alone whether tapping will start or stop recording until they read the
  inner text; an `.accessibilityValue("слушаю" / "остановлено")` belongs
  on the button.
- **SHOULD-FIX** `Dspeech/App/ContentView.swift:122` — error banner
  `Text(error)` carries identifier `error-banner` but no
  `.accessibilityLabel("Ошибка: \(error)")`. VO reads the raw message
  with no category prefix.
- **SHOULD-FIX** `Dspeech/App/ContentView.swift:141` — app title
  `Text("Dspeech")` is not marked `.accessibilityAddTraits(.isHeader)`,
  so VO rotor "Headings" cannot land on it.
- **SHOULD-FIX** `Dspeech/App/ContentView.swift:280` — `Image(systemName:
  "waveform")` in `PartialTranscriptCard` is decorative and not marked
  `.accessibilityHidden(true)`.
- **SHOULD-FIX** `Dspeech/App/ContentView.swift:334` — language-code
  badge `Text(segment.sourceLanguageCode.uppercased())` is read by VO as
  letter-by-letter "E-N". Needs an `.accessibilityLabel` resolved via
  `Locale.current.localizedString(forIdentifier:)` ("Английский").
- **SHOULD-FIX** `Dspeech/App/ContentView.swift:341` — `Text("VERIFY")`
  is the English literal; the rest of the surface is Russian. Add
  `.accessibilityLabel("Низкая уверенность, проверьте")`.
- **SHOULD-FIX** `Dspeech/App/ContentView.swift:351` — confidence
  percentage has no label context. VO reads "девяносто три процента"
  with no anchor. Add `.accessibilityLabel("Уверенность \(...)")`.
- **NICE** `Dspeech/App/ContentView.swift:237-249` — `LabeledContent`
  rows ("Язык по умолчанию", "Модель ASR", "Режим", "Целевой язык",
  "Провайдер", "Версия") have no identifiers. Future UITests covering
  Settings cannot anchor them.
- **SHOULD-FIX** `Dspeech/App/FirstRunView.swift:206` — `Text(card.title)`
  lacks `.accessibilityAddTraits(.isHeader)`.
- **NICE** `Dspeech/App/FirstRunView.swift:237` — duplicate label
  "Язык перевода" appears as both the section header `Text` and the
  `Picker` label, so VO announces it twice.
- **SHOULD-FIX** `Dspeech/App/FirstRunView.swift:259-269` — `pageDots`
  is `.accessibilityHidden(true)`, so a VO user has no idea where they
  are in the 3-card flow. Expose
  `.accessibilityLabel("Шаг \(cardNumber) из \(cardCount)")` somewhere
  in the visible card.
- **SHOULD-FIX** `Dspeech/App/SettingsSheet+Sections.swift:40` —
  selected-input `Image(systemName: "checkmark")` is the only signal of
  selection; the row lacks `.accessibilityAddTraits(.isSelected)`, so
  VO does not say "Selected" on the chosen input.
- **SHOULD-FIX** `Dspeech/App/SettingsSheet+Sections.swift:84` — level
  meter has `.accessibilityIdentifier` and `.accessibilityValue` but no
  `.accessibilityLabel("Уровень сигнала")`; VO reads the percentage with
  no anchor.
- **NICE** `Dspeech/App/AboutView.swift:38-40` — "Аудио" attribution row
  has no identifier sibling to `about-attribution-apple-speech` /
  `about-attribution-translation`. Future tests cannot pin it.

## 2. VoiceOver order / grouping

- **SHOULD-FIX** `Dspeech/App/ContentView.swift:332-355` — `badgeRow`
  combines source-lang chip, VERIFY chip, `Spacer`, and confidence
  percentage as four discrete VO stops. Wrapping the row in
  `.accessibilityElement(children: .combine)` would yield one coherent
  announcement: "Английский. Низкая уверенность, проверьте. 78
  процентов." instead of three disconnected hops.
- **NICE** `Dspeech/App/ContentView.swift:152-160` — `Toggle("Перевод")`
  sits inside the control bar in the *trailing* position but
  *precedes* (in source order) the segment cards. Since the toggle
  controls whether translation appears in every card below, that
  ordering is correct — flagged only as an explicit confirmation.
- **NICE** `Dspeech/App/SettingsSheet+Sections.swift:222-232` —
  `statusRow` switches on `status`. When status is `nil` ("Проверка
  доступности…") the row appears before the picker has been operated;
  no `.accessibilityElement(children: .combine)` on the parent Section,
  so VO reads the picker then the standalone status — acceptable but
  worth a one-line combine.

## 3. Dynamic Type — fixed-size fonts that break at AX5

All transcript content and onboarding copy uses
`.font(.system(size: N, ...))` literals. None scale with the user's
preferred content size, so an AX5 user reads the same point size as a
medium user. This is the single largest accessibility hole in the
codebase.

- **BLOCKER** `Dspeech/App/ContentView.swift:61` — empty-state text
  20pt (18pt landscape), primary onboarding-like guidance.
- **SHOULD-FIX** `Dspeech/App/ContentView.swift:142` — app title 28pt
  (22pt landscape).
- **SHOULD-FIX** `Dspeech/App/ContentView.swift:197` — privacy badge
  text 11pt (10pt landscape). Monospaced + tiny.
- **BLOCKER** `Dspeech/App/ContentView.swift:285` — partial transcript
  26pt (22pt landscape). Live ASR output is the product's reason to
  exist; not scalable.
- **BLOCKER** `Dspeech/App/ContentView.swift:363,370,378,383,391` —
  segment text (source + translation) ranges 22–30pt fixed. Worst case
  at AX5 the user is asked to read 30pt content while their preferred
  size is far larger. `.minimumScaleFactor(0.55/0.6)` only shrinks for
  narrow widths, never grows for Dynamic Type.
- **BLOCKER** `Dspeech/App/FirstRunView.swift:207` — onboarding card
  title 30pt fixed.
- **BLOCKER** `Dspeech/App/FirstRunView.swift:213` — onboarding card
  body 19pt fixed.
- **SHOULD-FIX** `Dspeech/App/FirstRunView.swift:202` — onboarding
  icon 64pt fixed (decorative, hidden — flagged only as it visually
  dominates the card and pushes scalable copy off-screen at AX5).
- **SHOULD-FIX** `Dspeech/App/AboutView.swift:87` — `LocalOnlyBadge`
  text 12pt fixed monospaced. Has a label so VO is fine, but visually
  unreadable at AX5.

`Section`/`LabeledContent`/`.font(.body|.footnote|.subheadline|.caption|
.headline)` instances elsewhere DO inherit Dynamic Type and are fine.

## 4. Fixed frames that clip at AX5

- **SHOULD-FIX** `Dspeech/App/ContentView.swift:101` — Start/Stop button
  `.frame(minWidth: isLandscape ? 110 : 130)` is a *min*, so the label
  can grow horizontally; but the vertical padding is fixed (10pt) and
  the inner `.font(.headline)` *does* scale, so at AX5 the headline
  overflows the capsule. Either drop the explicit `.padding(.vertical:
  10)` or let the capsule grow vertically with `Capsule()`'s natural
  size.
- **SHOULD-FIX** `Dspeech/App/ContentView.swift:165,172` — settings
  button uses a fixed circular frame `diameter: 32/36`. The contained
  icon is `.font(.system(size: 15/17, ...))` (also fixed), so circle
  size is consistent — but the system minimum touch target is 44×44pt,
  and this button is 32–36pt. **BLOCKER touch-target** for users with
  motor impairment.
- **SHOULD-FIX** `Dspeech/App/FirstRunView.swift:297` — primary button
  uses `.padding(.vertical, 16)` + a scalable headline inside. At AX5
  the label can overflow horizontally because `.frame(maxWidth:
  .infinity)` forces a single line. Add `.lineLimit(nil)` and
  `.fixedSize(horizontal: false, vertical: true)`.

## 5. Contrast / hardcoded color combinations

(All values eyeballed against the `Color.black → Color(red:0.03, g:0.06,
b:0.10)` gradient background.)

- **NICE** `Dspeech/App/ContentView.swift:62,115,155,191,353` —
  `.white.opacity(0.6–0.85)` for body and secondary copy. At opacity
  0.6 on near-black the effective contrast is ~7:1, which clears WCAG
  AA. No fix required, flagged only because the values are not pulled
  from a semantic palette.
- **SHOULD-FIX** `Dspeech/App/ContentView.swift:103` — listening state
  is communicated by background color alone (`Color.red.opacity(0.85)`
  vs `Color.cyan.opacity(0.85)`). Symbol changes (`stop.fill` vs
  `mic.fill`) and text changes ("Стоп" / "Старт") also encode the
  state, so this is acceptable — but the listening state is **not**
  exposed via `.accessibilityValue` (cross-ref §1), so a VO user
  depends entirely on the text label.
- **SHOULD-FIX** `Dspeech/App/ContentView.swift:194-198` — privacy
  badge tint is `.green` (LOCAL) vs `.orange` (CLOUD). For
  red-green-blind users the visual distinction is weak; the only
  textual cue is the 4–5 letter `badgeText`. The
  `.accessibilityLabel` ("Локальная обработка" / "Облачная обработка
  (с согласия)") at L206 mitigates for VO users, but sighted CVD users
  rely on text — small monospaced 10–11pt text. Pair with §3.
- **NICE** `Dspeech/App/ContentView.swift:343,351,367-385` — yellow
  `VERIFY` chip, white `.opacity(0.6)` confidence, white `.opacity
  (0.15)` divider rectangle — all decorative/state cues with text
  alternatives.
- **NICE** `Dspeech/App/FirstRunView.swift:296,299` — primary button
  is `.black` text on `Color.cyan` capsule. Contrast ~9:1, clears
  WCAG AAA.
- **NICE** `Dspeech/App/SettingsSheet+Sections.swift:225-229` —
  pack-status `Label` uses tint `.green/.cyan/.orange` paired with
  distinct SF Symbols and text. Color is redundant cue, not the only
  cue. Good.

## 7. Missing announcements for asynchronous state changes

No call to `UIAccessibility.post(notification: .announcement, argument:)`
exists anywhere in the codebase. The following transitions therefore
land silently for VoiceOver users:

- **BLOCKER** `Dspeech/App/ContentView.swift:121-127` — error banner
  appears when `liveViewModel.lastErrorMessage` flips from `nil` to a
  message. No announcement is posted. A blind user sees no transcript
  arrive and gets no spoken hint that the engine failed.
- **SHOULD-FIX** `Dspeech/App/ContentView.swift:80-81` — partial
  transcript appearing/disappearing is a meaningful event. At minimum,
  initial appearance should be announced
  ("Распознавание идёт"); steady-state updates should NOT be
  re-announced (would be noisy).
- **SHOULD-FIX** `Dspeech/App/LiveTranscriptionViewModel.swift:54-56`
  — `case .status(let newStatus)` transitions (`.ready` →
  `.listening`, `.listening` → `.stopped`, `*` → `.failed`) are stored
  on the view model and rendered by `ContentView`, but never posted as
  accessibility announcements. The user-facing announcement plumbing
  belongs at the view layer (which `ContentView` is), not on the view
  model itself.
- **SHOULD-FIX** `Dspeech/App/SettingsSheet+Sections.swift:256-269` —
  pack download completion (`status` flips to `.installed`) and
  failure (`errorText` populates) both happen silently. The CTA
  disappearing and a new status label appearing is a major UI change.
- **SHOULD-FIX** `Dspeech/App/SettingsSheet+Sections.swift:86-90` —
  the "Низкий уровень" warning appears when `level < 0.15`; no
  announcement, no `.accessibilityElement(children: .combine)` on the
  meter+warning pair.
- **SHOULD-FIX** `Dspeech/App/SettingsSheet+Sections.swift:127-134` —
  audio-route change handler (`observeRouteChanges`) silently reloads
  the input list and may flip `currentInputID`. For a user who just
  plugged in a headset, an announcement
  ("Новый вход: \(input.displayName)") would be the difference between
  knowing the headset took over vs. not.
- **NICE** `Dspeech/App/FirstRunView.swift:218` — last-card transition
  ("Начать" button appears, language picker appears) happens silently.
  Since the user just tapped "Далее", focus management is more
  important than an announcement.

## 8. Cross-cutting smells

- Every fixed-pt `Text` in §3 is grep-able with
  `rg "\.font\(\.system\(size:" Dspeech/App` — 19 hits. Fixing them
  collectively is a single mechanical pass (`.system(.body, design:
  .rounded)` etc.).
- Every `Image(systemName:)` in the surface is grep-able with
  `rg "Image\(systemName:" Dspeech/App` — 7 hits. A second mechanical
  pass auditing each for `.accessibilityHidden(true)` (decorative) vs
  `.accessibilityLabel(...)` (informational) covers §1's icon items.
